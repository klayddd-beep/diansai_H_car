## 零、总原则：坐标一律用「场地系 dm」

- 原点 = 题目图 1 **巡防区左下角**，右上角 = (48, 40)，单位 **dm**。
- 跨机传的所有坐标都是这个系，**两边收到即可直接用，不做任何二次换算**。
- 各自的 Cartographer `map` 系与场地系的标定各自负责：
  - 无人机侧：`fire_control_pkg` 的 `home_field_x_dm / home_field_y_dm / field_yaw_offset_deg`
  - 消防车侧：`fire_params.yaml` 的 `arena_origin_map_x_m / y_m / yaw_deg`

> ⚠️ **两边必须标到同一个物理原点**（同一个场地左下角）。任一侧标偏，火源坐标传过去就是错的，
> 车会开到错误位置。建议赛前用同一个基准点、同一把尺子，两边各标一次后互相验证：
> 把飞机停在某个已知格点，看车端显示的坐标对不对。

## 二、通道 1：火源坐标上报（机 → 车 :8889）

车端 `fire_event_bridge` 已按此格式实现，并会丢弃长度、magic、序号或坐标非法的数据包。

小端定长 **16 字节**：

```
偏移  类型      字段        说明
0     uint16   magic       固定 0xFC11
2     uint16   seq         递增，接收端按 int16 差值判新旧（车端已实现）
4     float32  x_dm        火源 x，场地系 dm
8     float32  y_dm        火源 y，场地系 dm
12    float32  reserved    置 0
```

行为约定：

- 无人机**抛完灭火包后**立刻发，**同一个 seq 连发 5 次**做丢包冗余。
- 车端按 seq 去重 —— 这一点你们现有代码已经做了，重复包不会触发多次出发。
- 全场只报**一次**（三个图案里只有目标是红色，无人机处理完即不再上报）。

## 三、通道 2：任务状态回传（车 → 机 :8890）

车端 `fire_event_bridge::reply()` 把 `/fire_mission_status`
的字符串原样 sendto 到「最近一个发火源包的 IP」的 8890 口。无人机侧会 bind 8890 接收。

请**确认**取值集合仍是这几个（无人机侧按这个表解析，用于在机上状态里显示车干到哪了）：

```
ready / enroute / extinguishing / returning / done / failed:<reason>
```

> ⚠️ **注意这条通道的一个隐含行为**：你们的 `reply()` 是发给「最近一个发过火源包的 IP」，
> 所以在无人机上报火源之前，`have_sender_` 一直是 false，**机端收不到任何状态**。
> 这不是 bug，但别误以为一上电就双向通了。如果希望全程可见车端状态，
> 需要改成从参数里读固定的无人机 IP——不过这对拿分没影响，优先级很低。

建议（不强制）：`done` 和 `failed:*` 这两个终态**连发 3 次**，因为它们是一次性事件，
丢了机端就永远不知道车完成了。中间态是周期性的，丢一两个无所谓。

## 四、通道 3：遥测（机 → 车 :8892）

**这条对应基本要求第 3、4 项，共 16 分。**

小端定长 **32 字节**：

```
偏移  类型      字段          说明
0     uint16   magic         固定 0xF14E
2     uint8    type          固定 1 (TELEMETRY)
3     uint8    phase         无人机任务阶段，见下表
4     uint16   seq           递增，按 int16 差值判新旧，丢弃乱序旧包
6     uint16   reserved16    置 0
8     uint32   stamp_ms      发送方单调时钟 ms。**不要做跨机时钟同步**，
                             判超时只用「收包本地时间」
12    float32  x_dm          无人机位置 x，场地系 dm
16    float32  y_dm          无人机位置 y，场地系 dm
20    float32  distance_dm   累计巡逻里程 dm
24    float32  height_dm     离地高度 dm
28    uint32   reserved32    置 0
```

C 结构体（加 `__attribute__((packed))`，`static_assert(sizeof(...) == 32)`）：

```cpp
struct __attribute__((packed)) FireLinkPacket {
  uint16_t magic;       // 0xF14E
  uint8_t  type;        // 1=TELEMETRY, 3=CAR_START
  uint8_t  phase;
  uint16_t seq;
  uint16_t reserved16;
  uint32_t stamp_ms;
  float    x_dm;
  float    y_dm;
  float    distance_dm;
  float    height_dm;
  uint32_t reserved32;
};
static_assert(sizeof(FireLinkPacket) == 32, "must be 32 bytes");
```

Python 解析格式串（自测用）：`"<HBBHHIffffI"`，`struct.calcsize` 必须等于 32。

### phase 取值

| 值 | 含义 | 值 | 含义 |
|---|---|---|---|
| 0 | 待命（等按键） | 6 | 抛包 |
| 1 | 起飞 | 7 | 恢复巡逻 |
| 2 | 巡逻 | 8 | 返航 |
| 3 | 接近火源 | 9 | 降落 |
| 4 | 降高 | 10 | 完成 |
| 5 | 悬停 | 255 | 未知 |

### 发送频率

题目原文是「巡逻期间无人机每秒向消防车发送 1 次位置坐标信息」，故**默认 1Hz**。
实现中从 Cartographer TF 可用起就发送，包括等待启动阶段；这样车端在按键前即可确认遥测链路正常。
无人机侧有参数 `telemetry_rate_hz` 可调高（多发不违规、显示更跟手），联调时可以先开到 5Hz
看链路稳不稳，正式测试再决定。

### 车端接收侧实现

1. bind UDP 8892，按 20ms 周期非阻塞轮询。
2. 校验 `长度 == 32`、`magic == 0xF14E`、type 和浮点字段，**按 seq 用 int16 差值判新旧**，丢弃乱序旧包。
3. 发布 `/drone_telemetry = [x_dm,y_dm,distance_dm,height_dm,phase,seq]` 供显示节点订阅。
   UDP 包内的 `stamp_ms` 保持不变，但不转成 float32 ROS 数组字段，避免精度损失后被误用于延迟测量。

显示部分见下一节，那是独立的一大块。

## 五、通道 4：按键启动（车 → 机 :8893）

**这对应基本要求第 2 项「在消防车上按键启动无人机」。**

同一个 32 字节结构体，只是字段填法不同：

```
magic       = 0xF14E
type        = 3   (CAR_START)
phase       = 255
seq         = 递增
stamp_ms    = 发送时刻
x_dm / y_dm / distance_dm / height_dm = 0
reserved32  = 0
```

目标地址固定为 `192.168.10.197:8893`，由车端 `drone_ip` 和 `start_udp_port` 参数配置；不使用 DHCP 地址学习或子网广播。

行为约定：

- 按键按下 → **连发 5 次**（间隔几 ms 即可），因为这是一次性事件，丢了飞机就不起飞。
- 无人机侧**只认第一次**，后续重复包全部忽略，不会重复触发。所以你们连发多少次都安全。
- 按键去抖由车端负责。飞机起飞后再按无效。

> 硬件上按键接哪个 GPIO / 用什么方式触发由你们定，飞机侧只关心这个 UDP 包。
> 如果一时接不上物理按键，先做成一个 ROS topic 或命令行触发也行，联调不受影响。

## 六、消防车显示器

**对应基本要求第 3、4 项，共 16 分。这是车端工作量最大、也最值钱的一块。**

前提：车上自带屏幕，直接在车载板子上开一个**全屏窗口**显示，不要用笔记本远程看
——题目原文是「消防车上显示器」，远程画面大概率不认。

### 必须显示的内容

| # | 内容 | 数据来源 | 对应要求 |
|---|---|---|---|
| 1 | 无人机**实时位置坐标**（数字，dm） | 遥测包 `x_dm / y_dm` | 基本要求 3 |
| 2 | 无人机**巡逻航迹曲线** | 历史遥测点连线 | 基本要求 4 |
| 3 | **累计巡逻里程**（数字，dm） | 遥测包 `distance_dm` | 基本要求 4 |
| 4 | 火源地点坐标（收到后常驻显示） | `/fire_event` | 发挥 2/3 |

> **里程直接用包里的 `distance_dm`，不要自己拿收到的点去积分。**
> 机端是用 Cartographer 位姿增量（约 20Hz）算的，你按 1Hz 采样点连线积分会显著偏小
> （拐弯处尤其）。想对照的话可以两个都显示，但报给评委的以包里的为准。

### 航迹曲线的具体要求

评委实际看的就是这张图，光有一条线和画在场地图上，观感差很多：

- **按比例画整个巡防区**：48dm(宽) × 40dm(高)，长宽比必须对，别拉伸。
- **底图要有**：
  - 六个白色街区矩形——直接读 `fire_params.yaml` 顶部共享的 `obstacles_dm`
  - 左下无人机起降区（黑色，7×7dm）
  - 消防车出发区（红色，`home_x_dm/home_y_dm` 附近）
- **坐标方向**：原点在**左下角**，x 向右，y 向上。屏幕像素 y 轴是向下的，
  记得翻转，否则整张图上下颠倒。
- **航迹**：历史点按时间顺序连成折线，实时追加；当前位置用一个明显的标记（圆点/十字）。
- **火源**：收到火源坐标后在图上打一个显眼的 ★，并标注坐标数字。
- **消防车自己的位置**：如果车端 TF 可用，一并画出来（不是要求项，但演示效果好很多）。

### 文字区

字号要大——评委是**站着看**的，别用默认小字。至少包含：

```
无人机  (12.4, 20.0) dm   高度 18.2dm   阶段: 巡逻
里程    124.5 dm
消防车  enroute
火源    (30.0, 25.0) dm
链路    正常 / ⚠ 离线 3.2s
```

- **`phase` 和 `height_dm` 必须显示**。比赛时你们只能看这块屏，这两个是判断飞机是否正常的
  唯一手段——`phase` 长时间不变基本就是飞机卡住了，得准备停机。
- **链路健康**：超过 3 秒没收到遥测包就显示红色「离线」和断连时长。
  UDP 没有连接概念，不显示这个的话链路断了你们会毫无察觉，还以为飞机在正常巡逻。

### 当前技术实现

当前使用 Python、Pillow 和 OpenCV 实现全屏界面，包含自适应布局、中文字体检测、地图卡片、状态徽标和分组遥测卡片。仍需配置桌面会话开机自启，比赛现场不应依赖手工启动。

### 验收标准

用 §八 的假遥测脚本喂数据，屏幕上应该看到：

1. 一个点沿 y=4 那条带从左往右移动，航迹连成一条直线
2. 里程数字线性增长
3. 停掉脚本 3 秒后，链路状态变红显示「离线」
4. 整张场地图长宽比正确、街区位置与实际场地对得上、原点在左下

## 七、一次完整测试的时序

```
1. 两板上电，各自 Cartographer 起来，TF 正常
2. [车] 操作员按下启动键
   → 车发 CAR_START ×5 → 机 :8893
3. [机] 收到 → 起飞到 18dm → 开始蛇形巡逻
   → 持续每秒发 TELEMETRY → 车 :8892（等待启动阶段也发送）
   → 车端屏幕开始画航迹曲线、刷坐标和里程
4. [机] 巡逻中下视相机发现红色火花图案
   → 接近确认 → 机上 LED 报警 → 降到 10dm → 悬停 3s → 抛灭火包
   → 发火源坐标 0xFC11 ×5 → 车 :8889
5. [车] 收到火源坐标 → 显示 → BFS 绕开街区 → 开到照射位并对准火源
   → 激光照射 2.1s → 返回红色出发区
   → 全程回传状态字符串 → 机 :8890
6. [机] 抛完继续巡逻 → 覆盖完成 → 返航 → 降落在起飞区内
   → phase 变 10(完成)
7. 发挥部分限时 360s
```

## 八、联调自测（不需要真飞机 / 不需要真车）

无人机侧仓库里有 `scripts/fire_link_test.py`，它**站在消防车的位置**收发包，
可以直接当车端接收逻辑的参考实现。

```bash
# 在车板上跑（或本机跑，把 --drone-ip 换成 127.0.0.1 自环）
./fire_link_test.py --drone-ip 127.0.0.1 --start-after 3
```

它会：bind 8892 收遥测并打印位置/里程/高度/阶段，`--start-after` 秒后向飞机 8893 连发启动包。
默认端口已按本文档设好，不需要额外覆盖。

**做任务 1（按键启动）时可以这样自测**，完全不依赖飞机侧改完：
车端按键逻辑写好后，让它发包到 `127.0.0.1:8893`，同时本机跑
`./fire_link_test.py --drone-ip 127.0.0.1` 之外再起一个 socket 收 8893 验证包对不对；
或者更简单——直接用本脚本的 `make_start_packet()` 当标准答案比对字节。

反向的最小验证（车端写好后，不用飞机）：

```bash
# 假装飞机在发遥测
python3 -c "
import socket,struct,time
s=socket.socket(socket.AF_INET,socket.SOCK_DGRAM)
for i in range(100):
    s.sendto(struct.pack('<HBBHHIffffI',0xF14E,1,2,i,0,int(time.monotonic()*1000)&0xFFFFFFFF,
             10.0+i*0.4,4.0,i*4.0,18.0,0),('192.168.10.113',8892))
    time.sleep(1)
"
```

车端屏幕应该看到一个点沿 y=4 那条带从 x=10 往右走，航迹画成一条直线，里程线性增长。
